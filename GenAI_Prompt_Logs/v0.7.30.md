## Security hardening changes

### 1. Added explicit extension-page CSP

Added this to `manifest.json`:

```JSON
"content_security_policy": {
  "extension_pages": "script-src 'self'; object-src 'none'; frame-src 'self'; connect-src 'self'; img-src 'self' data:;"
}
```

Why:

- `script-src 'self'` keeps scripts limited to packaged extension files.
- `object-src 'none'` blocks plugin/object/embed surfaces.
- `frame-src 'self'` limits embedded frames to extension-owned pages.
- `connect-src 'self'` prevents accidental remote fetch/XHR/websocket paths from extension pages.
- `img-src 'self' data:` keeps images local/extension-controlled.

### 2. Hardened bookmark URL opening

Added centralized URL filtering before extension-initiated opens.

SBM now only opens these bookmark URL schemes via extension APIs:

```JavaScript
http:
https:
ftp:
```

This applies to:

- row open / keyboard Enter
- Details pane “Open Bookmark”
- context menu “Open”
- open all in current window
- open all in new window
- open all in private window
- open all in tab group
- folder open-all
- multi-selection open-all

Why:

- Bookmark titles and URLs are user-controlled data.
- `chrome.tabs.create()` and `chrome.windows.create()` are sensitive because the extension actively tells Chromium where to navigate.
- Blocking unsupported schemes before calling those APIs reduces exposure to active or unusual schemes such as `javascript:`, `data:`, `blob:`, `file:`, and malformed URLs.

Existing bookmarks with unsupported schemes are not deleted or edited; SBM just refuses to open them through extension-initiated actions.

### 3. Added centralized safe URL helpers

Added helpers in `manager.js`:

```JavaScript
safeBookmarkOpenUrl()
safeBookmarkOpenUrls()
bookmarkOpenBlockedMessage()
```

Why:

- Avoids scattered ad-hoc URL checks.
- Makes future review easier: all extension-initiated bookmark opening now routes through one policy.
- Keeps context menus, folder open-all, multi-select, row open, and Details open consistent.

### 4. Hardened embedded Options iframe

The in-manager Options iframe now sets:

```JavaScript
frame.referrerPolicy = "no-referrer";
frame.sandbox = "allow-forms allow-same-origin allow-scripts";
frame.src = chrome.runtime.getURL("options.html?embedded=1");
```

Why:

- The iframe source remains a packaged extension page.
- `no-referrer` avoids leaking the manager page URL as referrer metadata.
- The sandbox limits iframe capabilities while still allowing the options page to run scripts and use its form controls.

### 5. Added error handling around the default bookmarks manager action

`openDefaultBookmarksManager()` now catches failures from:

```JavaScript
chrome.tabs.create({ url: "chrome://bookmarks/" })
```

Why:

- It prevents unhandled promise rejections.
- If Chromium refuses the navigation for any reason, the user gets a visible error instead of silent failure.

### 6. Normalized service-worker storage reads

In `service_worker.js`, manager tab IDs and the `MultipleInstancesAllowed` setting are now normalized before use.

Why:

- `chrome.storage.local` / `chrome.storage.session` contents should be treated as untrusted or at least mutable.
- The service worker now filters tab ID lists to integers and deduplicates them.
- `MultipleInstancesAllowed` is used only if it is actually a boolean.
