Moderately easy if we keep it simple and build on CSS variables. Harder only if you want a full theme editor with import/export and per-element customization.

For SBM specifically, I’d estimate:

**Basic theme support: low difficulty**

- Add a setting like `ThemeMode: "system" | "light" | "dark"`.
- Define colors through CSS custom properties in `:root`.
- Apply a class or `data-theme` attribute on `document.documentElement`.
- Respect `prefers-color-scheme` when set to `system`.
- Add Options UI dropdown.
- Add locale strings.
- Update `manager.html`, `options.html`, `about.html`, and likely shared CSS.

This is probably a clean v0.8.x feature.

**Preset themes: medium difficulty**

- Add several named themes, for example:

    - System
    - Light
    - Dark
    - High Contrast
    - Sepia / Warm
- Store selected theme in settings.
- Keep theme definitions centralized.
- Make sure tree rows, selected rows, drag targets, buttons, dialogs, context menus, details pane, warnings log, About page, and Changelog modal all remain readable.

This is still very manageable.

**Custom user themes: medium-high difficulty**

- Add color pickers in Options.
- Validate color values.
- Store custom theme object.
- Prevent unreadable combinations or provide reset buttons.
- Possibly add import/export.
- More testing needed.

This is where complexity grows.

The best implementation path would be:

1. First pass: refactor existing colors into semantic CSS variables, such as:

```CSS
--sbm-bg;
--sbm-surface;
--sbm-surface-raised;
--sbm-text;
--sbm-text-muted;
--sbm-border;
--sbm-accent;
--sbm-selection-bg;
--sbm-selection-text;
--sbm-danger;
--sbm-warning;
```

1. Add a tiny theme module, probably `theme.js`, used by manager/options/about pages.
2. Add settings:

```JavaScript
ThemeMode: "system"
ThemeName: "default"
```

1. Implement light/dark first.
2. Add more presets only after the variable model proves stable.

Main risks:

- SBM likely has many one-off color values in CSS right now, so the first pass is mostly careful CSS cleanup.
- Selection/focus/hover states need extra attention for accessibility.
- The embedded About/Options/Changelog views should use the same theme variables so they do not look detached.
- If you support system theme changes live, pages need to listen for `prefers-color-scheme` changes or reapply theme on focus/settings changes.

My recommendation: start with **v0.8.8: theme foundation + System/Light/Dark** only. Avoid custom colors until the app’s UI variables are stable.

## Summary of changes

### Theme foundation added

Added initial support for:

- **System**
- **Light**
- **Dark**

The new setting is available under:

`Options > User Interface > Theme`

Default is:

```JavaScript
ThemeMode: "system"
```

### Added shared `theme.js`

The new `theme.js` module handles:

- Theme setting normalization
- Applying theme attributes to the document root
- Resolving System mode to the current browser/OS color scheme
- Reacting to `prefers-color-scheme` changes when System mode is active

Stored theme values are constrained to known safe enum values only:

```JavaScript
"system" | "light" | "dark"
```

This avoids allowing arbitrary stored values to become CSS or DOM state.

### Added shared `theme.css`

Refactored page colors into semantic CSS variables, including variables for:

- background
- surface
- pane background
- text
- muted text
- borders
- accent color
- selection states
- overlays
- shadows
- links
- warnings
- code blocks

This now gives SBM a proper foundation for later custom themes without needing to rewrite the UI again.

### Applied theming across pages

Theme support is now used by:

- `manager.html`
- `options.html`
- `about.html`

Each page loads `theme.css`, and each relevant controller applies the current theme through `theme.js`.

### Security hardening considerations

- Theme values are normalized before use.
- No user-provided CSS colors are accepted yet.
- No `innerHTML` was added.
- No remote assets, scripts, or theme files were added.
- Theme application uses fixed `data-*` attributes and `color-scheme`, not arbitrary style injection.

### Performance considerations

- Theme changes are applied by changing document-level attributes/CSS variables.
- No repeated DOM walks are needed for theme application.
- System theme changes use one `matchMedia` listener per page.
- Existing UI rendering paths are not reworked or made heavier.

### Documentation

Added documentation comments to the new theme helper functions and updated related comments where theme handling was added.
